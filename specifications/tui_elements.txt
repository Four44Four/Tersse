 - Base element has:
    - Absolute position or parent element and relative position
    - Fixed width
    - Solid bg color
    - Text display with fg color
    - Can have fixed height and is scrollable by Alt/Meta + Up/Down keys when focused OR can have height that always conforms to text wrapped height
       - Fixed height will cause all text that overflows the element to be hidden
          - Scrolling with Alt/Meta + Up/Down keys when focused will hide the top most line and reveal the next hidden line until the end of the element is scrolled to
    - Supports composition pattern using any combination of the following properties:
       - Can have a callback function that runs (synchronously on UI thread) when Space or Enter key is pressed while the element is focused
       - Can have a text input property that overrides keybinds and allows behavior according to ./specifications/tui_element_text_input_property.txt when it is focused
          - Without this property:
             - The element's text content is not mutable through user keyboard input, but can still be updated through API functions to directly change the text display content
    - Element properties can be dynamically modified (in the context of the UI thread) to change their function
       - EXAMPLE: 
          - An element can behave like a static text display, but be later changed to become a button that responds to Space/Enter key input when focused, and the button can also be made into a text input that can support keystrokes and left/right arrow keys to add/remove text
 - Can be focused by navigating through page by pressing arrow keys
 - Has a fixed order of focus cycling explicitly defined
    - Similar to tab navigation order in webpages
    - Up/Right cycle to next element
    - Down/Left cycle to previous element
 - Do not render TUI elements beyond terminal bounds
 - Cut off parts of multi-line TUI elements that exceed terminal height
 - Cut off parts of TUI elements that exceed terminal width
 - Can be created with a parent element:
    - Will be positioned relative to parent element's Left, Right, Top, or Bottom sides
    - Default positions relative to parent element (location is {0, 0}):
       - Left -> child element's top left corner will start at the same Y position as the parent element, but X position will be the parent element's top left corner minus the width of the child element
       - Right -> child element's top left corner will start at the same Y position as the parent element, but X position will be the parent element's top left corner plus the width of the parent element
       - Top -> child element's top left corner will start at the parent element's Y position minus the child element's height, but the X position will be the same as the parent element's top left corner
       - Bottom -> child element's top left corner will start at the parent element's Y position plus the parent element's height, but the X position will be the same as the parent element's top left corner
    - `location` parameter will offset the child element further based on the above default locations
    - If parent element is removed:
       - Child elements are removed too
    - If parent element is moved:
       - Child elements will be moved by the same amount
 - If no parent element is supplied:
    - Will be positioned relative to the top left corner of the terminal
    - Default position will be at the top left corner (location {0, 0})
    - `location` parameter will offset element from the top left corner
 - Elements cannot overlap:
    - If an element is made with a location that causes it to overlap with another non-parent element, push either the lower element down and push all elements lower than it down enough so that there are no overlapping elements
 - Elements are to be stored in a Hashmap with ID as the key and the element as the value
    - Elements must be insertable, accessible, and deletable in O(1) time complexity
 - A secondary focus order sorted tree is to be maintained pairing focus_number float to element ID, sorted by focus_number float
    - Insertion, deletion, and arbitrary access from focus_number must be in O(log(n)) time complexity
    - Shifting focus between adjacent elements will traverse in-order in this tree to the next/previous pair
 - Newly created elements will generate a usize ID from an auto-incrementing global ID counter that can be used to uniquely identify them
    - If this overflows, it goes back down to 0 and repeats
    - Whenever an ID is generated:
        - If that ID already has an element associated with it
           - Try the next ID
           - Keep incrementing until an ID that doesn't have an associated element is foreground
        - Else:
           - Create the element and insert it into the element storage data structure normally
 - All ID related logic should be private to internal API only
    - Library API user should never manually supply or mutate an element's ID
